# EyeFlow engineering handbook

## Design goals

EyeFlow should feel calm, fast, and trustworthy during an eight-hour clinic shift. Desktop workflows are primary, responsive behavior is required, and every core action must remain keyboard accessible.

## Architecture

The repository is a pnpm/Turborepo monorepo. The web application uses TanStack Start so server functions and routes can evolve together while preserving API-first domain boundaries. Shared packages may expose source TypeScript to the application; each package still owns its public exports and type-checks independently.

Features should follow a vertical-slice shape:

```text
feature/
  components/
  server/
  schemas/
  queries/
  feature.test.ts
```

Only create layers that have behavior. Avoid generic repositories, service wrappers, and index barrels that merely forward implementation details.

## Domain conventions

- IDs are UUIDs generated by PostgreSQL.
- Timestamps are stored with time zone and treated as UTC at boundaries.
- Monetary amounts use `numeric(12,2)` in PostgreSQL and cross boundaries as decimal strings or validated minor units.
- Department visibility and mutation permissions are independent capabilities.
- EyeFlow has two application roles: `admin` and `user`.
- Users may edit current clinic-day collections; only administrators may edit historical collections.
- Daily targets are available to both roles. Weekly and monthly targets are administrator-only data.
- Historical edits require admin authorization and an audit reason.
- Database transactions protect multi-record financial changes.

## UI conventions

- Build from tokens in `apps/web/src/styles.css` and primitives in `packages/ui`.
- Visible focus, semantic labels, sufficient contrast, reduced-motion support, and touch-sized targets are mandatory.
- Every data view needs loading, empty, failure, and success states.
- Use restrained motion to communicate state; never make task completion wait for animation.
- Fixtures belong near their feature and must be clearly distinguishable from production data access.

## Security baseline

Better Auth is the authentication foundation. The web server owns the Drizzle adapter and `/api/auth/*` route. Authentication is not authorization: every private server function requires a session and a typed permission, while department-scoped mutations also check `user_department_access`. Secrets are injected at runtime, never bundled into the client.

Patient and payment mutations will require an append-only audit event. Logs must avoid patient names, payment details, tokens, and raw request bodies.

The FOSS EHR connector is a local browser-automation boundary. Its persistent profile contains a
sensitive authenticated session and must remain outside Git and container images. Synchronization
uses stable EMR patient and appointment identifiers, imports the minimum operational fields, and
does not persist phone numbers or clinical content. UI selectors are integration contracts and
must be covered by parser tests because the upstream product does not provide a supported API.
Only administrators may create or replace the browser session. Authenticated revenue users may run
a manual synchronization, and the dashboard schedules the same audited server operation while it is
open. Do not move browser credentials, cookies, or profile contents into application tables. A
headless worker with an encrypted persistent volume is required before enabling unattended sync in
Kubernetes.

## Testing strategy

- Vitest covers domain rules, schemas, and focused component behavior.
- Playwright covers critical workflows and accessibility-visible states.
- Database integration tests use an isolated PostgreSQL database.
- CI runs formatting/lint checks, type checking, tests, production build, and Chromium smoke tests.

## Delivery

The Docker image builds the selected web workspace and runs as an unprivileged user. Compose is for local development and evaluation. Kubernetes manifests provide a secure K3s-ready web workload but deliberately do not run production PostgreSQL; use a backed-up managed or operator-controlled database for production.

## Definition of done

A change is done when behavior, authorization, audit impact, tests, failure states, configuration, migration safety, documentation, and deployment impact have all been considered and the repository passes its verification commands.
